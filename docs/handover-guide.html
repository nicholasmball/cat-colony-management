Part 1 What each piece is — in plain English

The app is stitched together from a handful of separate online services, each doing one job. None of them are unusual — they're the standard, well-trodden tools for an app like this. Here's each one, what it does, and the plain-English version.

🟢 Vercel — where the app actually lives

Vercel is the host: the company whose computers run the website itself (the pages, buttons, and screens your volunteers tap). When someone opens the app on their phone, they're talking to Vercel. The app is built with Next.js (a popular way of building modern web apps); Vercel is the home built for running Next.js.

Plain version: Vercel is the shop floor — the place customers walk into. It serves the pages but doesn't itself store the cat records; it asks the database for those.

🗄️ Supabase — the database, the logins, and the rules

Supabase does three jobs at once:

  • The database — the permanent record of everything: colonies, cats, feeding schedules, incidents, notifications. (It's a Postgres database — a very standard, reliable kind.)
  • Logins (Auth) — it handles signing volunteers in, password resets, and the invite-only sign-up.
  • The access rules (RLS) — "Row-Level Security" is a built-in guard that makes sure each organisation only ever sees its own data, and each role (Admin / Caretaker / Feeder) can only do what it's allowed. These rules live inside the database, so they can't be bypassed by a bug in the app.

The shape of the database (its tables and rules) is defined by a set of migration files in the code — numbered 0001 through 0010. Running them in order on a fresh Supabase project builds an identical, empty database ready for data.

Plain version: Supabase is the filing cabinet, the front-door lock, and the rulebook all in one. Nick's project is reference ogdeoskhplqnguyacxyh; Street Cats will get a brand-new one of their own.

📷 Cloudflare R2 — where the cat photos are stored

Photos don't go in the database (they'd bloat it). They live in Cloudflare R2, a storage bucket — think of it as a private online photo drawer. The bucket is named scot-photos and is private: nobody can reach a photo by guessing its web address.

When the app needs to show a photo, it asks R2 for a short-lived signed link — a one-time web address that works for a few minutes and then stops working. That keeps the photos private while still letting the right people see them.

Plain version: R2 is a locked photo drawer. The app hands out a temporary key each time a photo needs showing, and the key expires on its own.

⏰ The automatic watcher — pg_cron, pg_net & Vault

Some alerts can only be noticed by a clock: "this colony's feeding window passed and nobody fed it," or "this cat hasn't been seen in a while." Nobody is sitting there watching, so the app has a built-in timer that checks for these on its own.

  • pg_cron is a scheduler that lives inside Supabase. It runs a small task every 15 minutes.
  • That task uses pg_net (a tool that lets the database call a web address) to ping the app at /api/cron/alerts — the page that runs the "missed feed" and "not seen" sweeps.
  • A second scheduled task pings /api/cron/email-digest once a day (around 14:00 UTC, i.e. mid-afternoon Portugal time) to send the daily summary email.
  • Vault is Supabase's secret-keeper. The timer has to prove it's allowed to trigger these — it carries a password (the CRON_SECRET) stored safely in Vault, and the app only runs the sweep if the password matches. (In the database, that stored password is named cron_alert_bearer.)
Plain version: a quiet caretaker who checks the colonies every 15 minutes and posts a daily summary — and shows a pass before they're let in, so randoms can't trigger it.

✉️ Resend — sending emails

The app sends a few kinds of email: a "you've been invited" message when an admin adds a volunteer, a gentle daily summary of non-urgent things, and password-reset emails. These go out through Resend, a service that specialises in sending app email reliably (so it doesn't land in spam).

There are really two email paths, and it's worth knowing the difference:

  • Invites & the daily digest — sent by the app's own email layer through Resend.
  • Password reset / confirm — sent by Supabase's login system, which is pointed at Resend using SMTP (the standard plumbing for sending mail), with branded templates so the emails look like Street Cats.
Important for handover Right now email runs on Nick's personal Resend account, sending from the domain buddha-balla.com. Part 2 swaps this to Street Cats' own Resend account and domain. Good news: it's purely a settings change — no code edits needed.

📱 PWA / offline — and English + Portuguese

The app is a PWA ("Progressive Web App") — a website that can be added to a phone's home screen and behaves like a normal app, including working without signal in the field. Updates made offline queue up and sync when the phone gets a connection again. This offline behaviour is controlled by a setting called NEXT_PUBLIC_SW_ENABLED (currently on), with an emergency off-switch called NEXT_PUBLIC_SW_KILL if it ever needs disabling fast.

Everything is available in English and (European) Portuguese, and the app remembers each person's choice.

Plain version: it installs like an app, works off-grid, syncs later, and speaks both languages.

  • Resend — sending email.

Part 2 The migration runbook — click by click

Do these in order. The golden rule: build the new home first, copy the live data across, point the door at the new home, then — only once it's confirmed working — take down the old one and rotate every password. Take your time; nothing here is reversible-by-accident if you leave Nick's accounts running until the very end.

Two facts to keep handy while you work The live Street Cats organisation inside the database has the id 46c8fdda-1a5e-481f-bc9e-4f98069afd91, and it holds the real data imported from the Wix site — 152 cats + 22 colonies + photos. Imported cats sit in an "Imported from website" holding colony for caretakers to reassign.

① Supabase — the new database, rules, timer & data

1a · Create Street Cats' Supabase project

  1. At supabase.com, sign in to Street Cats' new account and click New project. Pick the EU (e.g. Frankfurt/London) region — closest to Portugal. Set a strong database password and save it in the password manager.
  2. Once it's created, go to Project Settings → API and copy two values for later: the Project URL (becomes NEXT_PUBLIC_SUPABASE_URL) and the anon / public key (becomes NEXT_PUBLIC_SUPABASE_ANON_KEY). Also copy the service_role key (becomes SUPABASE_SERVICE_ROLE_KEY) — treat this one like a master key, never share it.

1b · Build the database shape (apply migrations 0001–0010)

The repo's supabase/migrations/ folder holds ten files, 0001_init.sql through 0010_recent_per_cat_views.sql. Applied in order on the fresh project, they recreate every table and the access rules (RLS).

  • Easiest: use the Supabase MCP / CLI's apply-migration in order — the same way they were applied originally (see your MEMORY note "Applying Supabase migrations").
  • Or by hand: Supabase dashboard → SQL Editor → paste each file's contents in numerical order (0001 first, 0010 last) and run each. Don't skip or reorder.

1c · Confirm the access rules (RLS) are intact

  1. In the dashboard go to Authentication → Policies (or Database → Tables) and check that the main tables show "RLS enabled" with policies present. The migrations create these, so if 0001–0010 all ran, they're there.
  2. Run Supabase's Advisors (Database → Advisors) and confirm there are no "RLS disabled on a public table" warnings.

1d · Move the live data across

There are two routes — pick one:

Route A — copy the existing data (recommended: keeps everything, incl. logins & history) Take a dump (a complete export) of Nick's project and restore it into the new one: 
# Export from Nick's project (connection string from its dashboard)
pg_dump "postgres://...OLD_PROJECT..." \
  --no-owner --no-privileges -Fc -f scot.dump

# Restore into Street Cats' new project
pg_restore --no-owner --no-privileges \
  -d "postgres://...NEW_PROJECT..." scot.dump
Do this after 1b so the shape exists, or restore data-only. Verify counts afterwards (see 1f). Confirm at migration time whether to dump schema+data or data-only, depending on exactly how the new project was built in 1b — easiest is a full pg_dump into a truly empty project and skip 1b.
Route B — re-import fresh from the Wix site Re-run the idempotent importer against the new project + new photo bucket: node scripts/import-from-site.mjs (it re-pulls 152 cats + 22 colonies + photos and is safe to re-run). This rebuilds the cats/colonies/ photos but not any feeding history, incidents, or volunteer logins created since launch — so for a live app that's been in use, Route A is the safe choice. Route B is the fallback if a dump/restore proves awkward.

1e · Recreate the two scheduled timers

The timers don't survive a data copy — recreate them on the new project. First make sure the two extensions are on (Database → Extensions → enable pg_cron and pg_net), then store the cron password in Vault and schedule both jobs.

  1. Store the password in Vault: dashboard → Project Settings → Vault → add a secret named cron_alert_bearer whose value is the same string you'll set as CRON_SECRET in Vercel (step ③). They must match exactly.
  2. Schedule the alert sweep (every 15 min) in the SQL Editor — point the URL at the new Vercel address: 
    select cron.schedule(
  'alerts-sweep', '*/15 * * * *',
  $$ select net.http_post(
       url := 'https://YOUR-NEW-APP.vercel.app/api/cron/alerts',
       headers := jsonb_build_object(
         'Content-Type','application/json',
         'Authorization','Bearer ' || (select decrypted_secret
           from vault.decrypted_secrets where name='cron_alert_bearer')),
       body := '{}'::jsonb); $$ );
  3. Schedule the daily digest (the email-setup doc uses 07:00; the live one runs ~14:00 UTC — pick the time Street Cats prefer, mid-afternoon Portugal is 0 14 * * *): 
    select cron.schedule(
  'email-digest-daily', '0 14 * * *',
  $$ select net.http_post(
       url := 'https://YOUR-NEW-APP.vercel.app/api/cron/email-digest',
       headers := jsonb_build_object(
         'Content-Type','application/json',
         'Authorization','Bearer ' || (select decrypted_secret
           from vault.decrypted_secrets where name='cron_alert_bearer')),
       body := '{}'::jsonb); $$ );
  4. Verify: after 15 minutes, check cron.job_run_details and net._http_response for a 200 reply. (Schedule these after Vercel is live in step ③ so the URL exists.)

1f · Set up login emails (Supabase Auth custom SMTP)

  1. Dashboard → Authentication → Emails → SMTP Settings → enable Custom SMTP.
  2. Point it at Street Cats' Resend (from step ⑤): host smtp.resend.com, port 465, username resend, password = a Resend API key. Set the sender to the verified Street Cats address.
  3. Paste the branded email templates (the design is in docs/supabase-reset-email.html) into Authentication → Emails → Templates.
  4. Under Authentication → URL Configuration, set the Site URL and Redirect URLs to the new app address, including …/auth/confirm so reset links land correctly.

1g · Verify counts

In the SQL Editor, confirm the data made it: you should see ~152 cats and 22 colonies for org 46c8fdda-1a5e-481f-bc9e-4f98069afd91. If those numbers match, the move worked.

② Cloudflare R2 — the new photo drawer

  1. In Street Cats' Cloudflare account, open R2 and click Create bucket. Name it scot-photos (keeping the same name means the stored photo links don't need rewriting). Keep it private.
  2. Create an R2 API token with Object Read & Write permission on that bucket. Copy the four values for Vercel later: Account ID (R2_ACCOUNT_ID), Access Key ID (R2_ACCESS_KEY_ID), Secret Access Key (R2_SECRET_ACCESS_KEY), and the bucket name (R2_BUCKET = scot-photos). The secret is shown once — save it now.
  3. Copy the photos across. Use an S3-compatible copy tool (e.g. rclone set up with both R2 accounts) to mirror every object from Nick's scot-photos bucket into the new one. Or, if you took Route B in 1d, the importer already re-uploaded the photos to the new bucket — in which case skip this.
  4. Only if you changed the bucket name: the cat photo links (photo_url values in the database) embed the bucket name, so they'd need updating with a one-off SQL update. Keeping the name scot-photos avoids this entirely — which is why we kept it.
  5. Verify: the new bucket's object count should match the old one.

③ Vercel — the new home for the app

  1. In Street Cats' Vercel account, click Add New → Project and connect the GitHub repository for the app. (Street Cats will need access to the repo, or it moved into their GitHub org.)
  2. Before the first deploy, open Settings → Environment Variables (set them for Production) and add every variable below.
  3. Deploy to Production. Vercel gives the project a …vercel.app address — note it; the timers (1e) and domain (④) point at it.

The full set of environment variables (the app's settings), with what each is:

Variable What it is / where it comes from
NEXT_PUBLIC_SUPABASE_URL New Supabase Project URL (step 1a)
NEXT_PUBLIC_SUPABASE_ANON_KEY New Supabase anon/public key (1a)
SUPABASE_SERVICE_ROLE_KEY New Supabase service_role key (1a) — master key, secret
R2_ACCOUNT_ID Cloudflare account ID (step ②)
R2_ACCESS_KEY_ID R2 token access key (②)
R2_SECRET_ACCESS_KEY R2 token secret (②) — mark Sensitive
R2_BUCKET scot-photos
CRON_SECRET A new long random string — must equal the Vault cron_alert_bearer (1e)
RESEND_API_KEY Street Cats' Resend API key (step ⑤)
EMAIL_ENABLED Exactly true to turn email on
EMAIL_FROM e.g. Street Cats of Tavira <no-reply@app.streetcatsoftavira.org>
NEXT_PUBLIC_SITE_URL The live app address, e.g. https://app.streetcatsoftavira.org
NEXT_PUBLIC_DEFAULT_LOCALE Default language, e.g. pt or en
NEXT_PUBLIC_SW_ENABLED true — keeps offline mode on (as in prod today)
NEXT_PUBLIC_SW_KILL Leave unset; the emergency off-switch for offline mode
About the "Sensitive" tick in Vercel Marking the R2 secret (and other keys) Sensitive means Vercel won't ever show the value back to you — safer, but it also means you can't pull it back later. So always copy these values from their original source (Cloudflare, Supabase, Resend), not from Vercel. Keep them in the password manager too.

④ Domain — point the web address at the new app

  1. In Street Cats' Vercel project → Settings → Domains, add app.streetcatsoftavira.org.
  2. Vercel shows a DNS record to add (usually a CNAME). Add it wherever Street Cats' domain DNS is managed (their registrar or Cloudflare). The apex marketing site stays on Wix and is untouched — this is only the app. sub-name.
  3. Wait for Vercel to show the domain as Valid / verified (DNS can take a little while). Then confirm NEXT_PUBLIC_SITE_URL in step ③ matches it, and that the timer URLs in 1e use it (or the …vercel.app address — both work).

⑤ Email — Street Cats' own Resend, off buddha-balla.com

  1. In Street Cats' new Resend account, verify the domain (add the SPF, DKIM and DMARC DNS records Resend shows) so email can send from e.g. app.streetcatsoftavira.org. Wait for "Verified".
  2. Create a Resend API key (shown once — copy it). Put it in Vercel as RESEND_API_KEY and set EMAIL_FROM to the verified sender (step ③), then redeploy.
  3. Point Supabase Auth's custom SMTP at this same Resend account (step 1f). This is what finally moves everything off Nick's buddha-balla.com.
  4. The "SMS for urgent alerts" option is post-launch and not part of this migration — it needs a paid provider plus Catherine's decision on storing phone numbers (see the channels note).

⑥ Handover hygiene — rotate secrets & hand over ownership

Once the new instance is confirmed working (do the smoke test below first), close the door on the old setup. Any password that ever passed through Nick's machine should be replaced with a fresh one that only Street Cats hold.

  1. Rotate every secret: generate a new Supabase service-role key, new R2 access keys, a new CRON_SECRET (update both Vercel and the Vault cron_alert_bearer so they still match), and a new RESEND_API_KEY. Redeploy after changing Vercel values. Because the R2 secret in Vercel is marked Sensitive, you can't read the old one back — that's fine, you're replacing it from Cloudflare anyway.
  2. Remove Nick's access: take Nick off the Supabase project, Cloudflare account, Vercel team, Resend account, and (once moved) the GitHub repo. Confirm Catherine / Street Cats are the owners, not just members, of every account.
  3. Decommission the old stack last: once everything runs on Street Cats' accounts and the smoke test passes, pause/delete Nick's old Supabase project, the old R2 bucket, the old Vercel project, and the buddha-balla.com Resend sender. Keep a final database dump as a backup until you're sure.

⑦ Final smoke-test — does the live instance actually work?

Open app.streetcatsoftavira.org on a phone and walk through it end to end. Tick each:

  • ☐ Log in as a caretaker test account.
  • ☐ See the colonies and cats — the 22 colonies and ~152 cats are there.
  • ☐ Photos load — open a cat with a photo; it appears (proves R2 + the signed links work).
  • ☐ Do a feeding update on a colony and confirm it saves and shows on Today.
  • ☐ Trigger / observe an alert — e.g. report an incident (fires instantly), or wait for the 15-minute sweep, and see it in the notifications bell.
  • ☐ Receive a notification + a branded email — invite a volunteer to your own address (branded invite arrives), and use Forgot password? to confirm the Supabase reset email comes through and the reset page works.

If all seven tick, the live Street Cats instance is standing on its own accounts end-to-end — and step ⑥ can be finished off with confidence.

That's the whole thing. Build the new home, copy the cats across, point the door at it, prove it works, then rotate the keys and hand over the deeds. Take it one numbered step at a time — there's no rush, and the old setup stays running as a safety net until the very last step. Thank you for everything you do for the cats. 🐾
