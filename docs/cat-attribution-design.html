Long address wraps (overflow-wrap: anywhere) rather than overflowing the 375px column or pushing the timestamp off-screen; full value stays available via the title attribute. The timestamp never gets orphaned because “· {when}” stays in the same paragraph flow. (Single-line ellipsis was rejected — it hides the most identifying part of an email and a 30-second field check shouldn’t need a hover.)

Component & token reuse

  • Line treatment: <p class="text-xs text-muted"> — copied from the incident header attribution (incidents/[incidentId]/page.tsx L217-220). Same size, same muted token, same · separator between name and time.
  • Email resolution: reuse the existing service-client lookup pattern (L126-165) — one getUserById per distinct id, no N+1; fall back to null → time-only line when the auth user is gone. Format with the same Intl.DateTimeFormat (org timezone, day/month/2-digit hour:minute) used on the incident page.
  • Confirmed-by line: tone-matches the incident resolution summary “✓ Resolved by … · …” (L246-252), rendered as the same plain muted line for visual calm.
  • Untouched: the status chip (statusTone / formatStatus from lib/cat-display.ts), the review note copy, and the Confirm/Reject CTA block (L183-207) all stay exactly as shipped. This change only inserts text lines.
  • Parity: feeders see the same attribution line as caretakers/admins — matching incident-detail behaviour where the reporter line is role-independent. Only the Confirm/Reject controls remain manager-gated.

Accessibility

  • Plain text, no colour-only meaning — the line reads identically in light/dark and to screen readers.
  • Muted timestamp uses the same --muted token already shipped AA-contrast on --surface; nothing new to audit.
  • Email is a real string, never a placeholder; the title attribute on long values is a supplement, not the only access path (the value also wraps and stays visible).

Email vs username (from Requirements): we display email because that is the shipped behaviour everywhere in the app today (incidents, comments, assignees). Introducing a human-friendly username is a separate product decision and out of scope for this additive line.
