Incident triage, resolution & list

Step 2 (UX) of the "Incident triage, resolution & list" task — the read + action + close half of incidents (the create form already shipped, see docs/incident-report-design.html). Step 1 (Requirements) confirmed: the data model and the managers update incidents RLS already exist (no triage migration); the only new migration is an append-only incident_comments table (comments are in scope). Lifecycle is open → in progress → resolved (+ reopen). The enum also carries closed, but Step 1 recommends — and this design adopts (a design-gate call) — collapsing resolved + closed into one terminal "Resolved" in the UI. Resolving requires a resolution note.

Permissions, stated once. admin / caretaker are managers — they triage, assign, transition status and resolve. A feeder is read + comment only and never sees status controls. The triage list and the sidebar "Incidents" item are manager-only; feeders reach a detail page only via a link (e.g. their own report's confirmation or the colony page) and there can only ever read and add a note. We render the difference by omission, never by disabled buttons — §6 shows the two variants side by side.
House style reused, no new primitive. Status & urgency pills are the exact StatusGlyph "icon + word, never colour alone" convention from app/app/today/page.tsx (fed/pending/missed → open/in progress/resolved). List rows mirror the Today / colony-detail tappable card rows; the Active/Done toggle is the segmented control from components/role-select-form.tsx; the resolve confirm reuses the styled components/confirm-button.tsx dialog; type glyphs come from the just-added IncidentTypeIcon / WarningIcon in components/icons.tsx. Tokens from lib/ui.ts (card · pill · btnPrimary · btnGhost · btnGhostDanger · input). Emoji are stand-ins for those SVGs so the mockup reads without assets.

1. Manager triage list — /app/incidents (admin/caretaker only)

A compact, thumb-reachable filter bar over a single sorted list. Sort: urgent + open first, then by time (newest). Filters (MVP set): an Active / Done segmented toggle (default Active = open + in progress), plus three chip toggles — Urgent only, by Colony (select inside a chip), Assigned to me. Each row is a full-width tap target → detail. Urgent rows are visually floated with a red left rail + red type tile (Gestalt/Fitts), never colour alone — every row pairs its status with an icon + word.

Why a terminal "Resolved" (no separate "Closed")
Design-gate: resolved + closed → one "Resolved". The enum keeps closed for the data layer, but two near-identical terminal states would force a volunteer to learn a distinction SCoT never asked for (Hick's Law / recognition). The UI exposes one done state ("Resolved"), reachable only with a note, and Reopen to undo. If SCoT later wants an archival "Closed", it's an additive lifecycle step, not a rewrite.
Status pills mirror Today. Open = neutral grey ●, In progress = amber ◐, Resolved = emerald ✓ — the exact tone family + icon-and-word rule as fed/pending/missed (StatusGlyph). Urgency is a separate red ⚠ badge so an urgent item is unmistakable independent of its status.

3. Resolve-with-note flow + validation

Resolving is consequential, so it's deliberate but not hostile: tapping Mark resolved… expands an inline note box in place (no full-screen modal). The note is required — Step 1's rule. Submitting empty shows an inline role="alert" error, marks the textarea aria-invalid, and keeps focus in the field; nothing is lost. This is the same "deliberate confirm" intent as components/confirm-button.tsx, expressed inline so the manager can see what they're resolving.

Block only the resolve. Empty note → role="alert" announces, textarea gets aria-invalid + red ring, focus stays in the field. The incident is untouched and the manager can still Cancel out. This is the one place in the whole feature where a field is genuinely required.

Other action states. Start and Reopen are single-tap status transitions (no note) and use the optimistic submit pattern. Resolve, while submitting, shows the button as aria-busy "Resolving…" and disabled — exactly the SubmitButton pending behaviour. Posting a comment shows the Post button as "Posting…" and disabled; on success the note appends to the thread and the textarea clears.

4. Comment thread — every role can add a note

Chronological notes (author + role + relative time), oldest first, with an Add a note textarea + Post button available to all roles including feeders — the one append-only incident_comments table. Status changes ("Marked resolved — …", "Reopened") are folded into the thread as dashed system entries so the whole timeline reads in one place. Empty thread shows a gentle prompt, never a blank void.

8. Accessibility notes (§)

  • § Status & urgency never colour-alone (WCAG 1.4.1). Every status pill carries a glyph + the word — Open ●, In progress ◐, Resolved ✓ — and urgency carries a ⚠ + "Urgent", reusing the StatusGlyph pattern from app/app/today/page.tsx. Urgent rows also get a left rail + tinted type tile, so prominence survives greyscale and colour-blindness.
  • § Labelled filter controls. Active/Done is a role="radiogroup" labelled "Show"; each chip is a real <button aria-pressed> with a visible label; the colony filter is a labelled <select>. The result count is a role="status" so filter changes announce.
  • § Resolve error announces. The empty-note error is role="alert" (assertive); the textarea gets aria-invalid + a red ring (not colour alone — it also shows the message and keeps focus).
  • § ≥44px targets. List rows ≥60px, segmented buttons ≥38px within a ≥44px group, chips ≥34px in a spaced row, every action button ≥44px, Post note ≥38px. Destructive-ish actions (Resolve) sit behind an explicit note step, harder to trigger than the constructive Start — per Fitts.
  • § Visible focus + keyboard order. Linear Tab: filter group → chips → each list row (one tab stop per row, the whole card is the link) → on detail: assignee → status buttons → resolve note → thread → Add-a-note → Post. The same focus:ring-accent/25 ring the design system already applies.
  • § Permission by omission, not disablement. Feeders get no disabled status controls to puzzle over — the Manage panel is absent and their Post-note button is the promoted primary action. No dead ends.
  • § Relative time in org tz. "12 min ago", "yesterday" rendered against the org timezone (org.timezone, like the Today page), with the absolute timestamp available on the detail header for precision.
