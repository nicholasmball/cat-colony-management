Feeding update — redesign

Step 2 (UX) of the "Redesign feeding-update page" task. Step 1 (Requirements) confirmed the problem and the fix: the current ✓ Fed toggle and the Save update submit are both full-width filled accent-blue, so users can't tell the selection toggle from the primary action — two equally-weighted blue buttons. Confirmed fix (Option A): replace the Fed toggle with a labelled segmented single-select "Was the colony fed?" control (✓ Fed / Not fed), green-on (emerald-600, the same family as the existing Seen sighting) / outline-off. Filled accent-blue is now reserved for the single primary CTA, Save update. Fed, flags and sightings become one consistent family of selection controls; none of them is accent-blue.

Presentation-only — data contract FROZEN. Hidden fed still posts "1"/"0" (defaults ON); flags problem/food_issue/danger and per-cat cat:<id> (seen|not_seen|concern|"") are unchanged; submitFeeding is untouched. Save update is the only type="submit"; every selection control (Fed, flags, sightings) is type="button". No new colours — reuse the existing emerald / amber / red and border-border tokens from lib/ui.ts + components/feed-form.tsx.

2. The "Was the colony fed?" segmented control — both states

A labelled section heading, then a 2-segment single-select. Exactly one segment is always selected; Fed is pre-selected by default (the common case). Each segment is ≥44px tall. The on-state deliberately matches the Seen sighting below it — same emerald-600 fill, white text — so the whole form reads as one family of selection controls (Gestalt similarity).

On-cue is never colour-alone. The selected segment carries weight (bold) + a fill + (for Fed) the ✓ glyph; the unselected one is a plain outline. "Not fed" selected uses a neutral muted fill (it isn't a positive state, so it must not borrow the emerald success colour — green stays reserved for "Fed" / "Seen").
Engineer note — reuses the existing tri-toggle, no new primitive. Render the two segments exactly like the sighting buttons in components/feed-form.tsx: a grid grid-cols-2 gap-2 wrapper, each a type="button" with min-h-11 rounded-lg border. Fed-on = border-emerald-600 bg-emerald-600 text-white font-semibold (identical to the Seen option's on class). Not-fed-on = a neutral selected style (border-muted bg-muted text-white / equivalent) so it doesn't read as success-green. Off = border-border text-foreground. The hidden <input name="fed"> stays and posts "1" when Fed is the chosen segment, "0" when Not fed — same state, same default (ON), same payload as today.

6. Accessibility notes (§)

  • § Segmented = single-select semantics. The two segments are a role="radiogroup" labelled "Was the colony fed?" (its visible heading), with each segment a role="radio" / aria-checked — exactly one checked at all times. (The current code uses a single aria-pressed toggle button; either a radiogroup or a 2-button aria-pressed pair is acceptable as long as the selected/unselected state is exposed and mutually exclusive. The heading must be programmatically associated as the group label.)
  • § Never colour-alone. The selected segment is signalled by fill + bold weight + the ✓ glyph (Fed) / fill + bold (Not fed), not hue. Legible in greyscale and to colour-blind users; mirrors the Step-1 rule already used for flags and sightings.
  • § "Not fed" doesn't borrow success-green. Emerald-600 stays reserved for the positive states (Fed / Seen). "Not fed" selected uses the neutral muted fill so colour meaning stays honest across the form.
  • § ≥44px targets. Both Fed segments, all three flag chips, every sighting button and Save are ≥44px tall (min-h-11+); the 2-up and 3-up grids hold full-width hit areas at 375px with no overflow.
  • § Visible focus. A focus-visible accent ring (≈3px, offset) on each control — the same visible focus the design system already applies — keeps the keyboard position obvious.
  • § Keyboard order. Linear top-to-bottom: Tab reaches Fed → flags → cats (Seen/Not-seen/Concern per cat) → Notes → Save update. As a radiogroup, the Fed segments are one tab stop with ←/→ to switch; Space/Enter selects. Save is the only control that submits.
  • § Visible label association. "Was the colony fed?" is a real on-screen heading tied to the group (not a placeholder); "Cats" and "Notes (optional)" keep their existing labels — recognition over recall, no hidden meaning.
  • § Error banner. The top-of-page banner keeps role="alert" so it announces on arrival; it sits above the form and never blocks Fed/flags/Save.
