i18n — EN | PT locale switcher UX Design · step 2 of 6

The translation framework (next-intl, cookie-based) is already wired: i18n/request.ts reads a locale cookie (default pt), and app/layout.tsx already sets <html lang={locale}>. The only UX surface to add is the EN | PT switcher in three placements, plus a side-by-side of how a real screen reads in each language. This deliverable reuses the existing segmented-control pattern.

1 · The switcher control — text labels, not flags

A segmented toggle, identical in look to the incidents Active / Done control (app/app/incidents/page.tsx) and the feed form's Fed / Not fed radiogroup (components/feed-form.tsx): a rounded-full bordered group with the selected segment filled in brand accent. We match the recently-fixed vertical centring (inline-flex items-center justify-center on every segment).

Why "EN | PT" text, not flags

  • Flags are countries, not languages. A Portugal flag excludes Brazilian readers; a UK/US flag is a poor stand-in for "English." Text avoids this misrepresentation.
  • Self-evident & legible at 375px. Two-letter codes stay crisp where a tiny flag glyph blurs and fails contrast.
  • Screen-reader friendly. "EN" / "PT" read cleanly; flag emoji read as country names.

Label decision: use the 2-letter codes EN / PT (uppercase) — not "English" / "Português". The full words make the segment too wide for the mobile account-menu row and the sidebar footer, and the codes are an internationally recognised, language-neutral convention. The control's accessible name carries the meaning for assistive tech (see §5).

Control choice: a labelled radiogroup of buttons, not a <select>. Two options sit better as a visible two-up toggle (recognition over recall, 44px targets, one tap) than a dropdown that hides the alternative behind a tap. Each segment is a real <button> inside a tiny <form> that POSTs to a server action — so changing language is a normal form submit that sets the cookie and refreshes (no client JS required; works offline-degraded and keyboard-first). A native <select> stays a valid fallback if the option set ever grows past ~3 languages.

3 · Switching feedback — cookie + refresh

Selecting a segment submits a tiny form to a server action that calls cookies().set("locale", …) then revalidatePath("/", "layout") (or redirect back to the current path). Server components re-render in the new locale and <html lang> updates. The choice persists across refresh and future visits via the cookie.

Tap EN (PT was selected)
→
Segment shows pressed + brief disabled/“busy” state during the round-trip
→
Server action sets locale=en cookie
→
Path revalidated; page re-renders in English, lang="en"
  • No jarring flash. It's a same-route refresh, not a navigation — the layout (logo, nav, scroll position) stays; only translated text swaps. Mark the pressed segment aria-busy="true" and dim it for the ~200–400ms round-trip so the tap feels acknowledged.
  • Announce the result. After re-render, an aria-live="polite" region states the active language in that language ("Language set to English" / "Idioma definido para Português") so screen-reader users get confirmation despite the full re-render.
  • Idempotent. Tapping the already-selected segment is a no-op (no needless refresh).

5 · Accessibility checklist

  • Reachable everywhere it's needed: pre-auth (home + login corner) and in-app (mobile account menu, desktop sidebar footer) — no signed-in-only language trap.
  • Named for AT: the radiogroup carries a bilingual accessible name (aria-label="Language / Idioma") or an associated visible "Language" label; each segment is a role="radio" with aria-checked.
  • Keyboard operable: Tab to the group, arrow-keys/Space/Enter to switch — standard radiogroup semantics; visible 3px focus ring on the focused segment.
  • Targets ≥44px (Fitts) in the primary control; the compact account-menu/sidebar variant is 40px tall but full-width-tappable within a generous row — still comfortably above the 24px AA floor.
  • Never colour-alone: the selected language is conveyed by the text label + aria-checked, not just the accent fill. Accent-on-white fill (#170df2 on #ffffff) and white-on-accent text both clear 4.5:1.
  • <html lang> matches the active locale (already wired in app/layout.tsx) so AT pronunciation and hyphenation are correct.
  • No hover-only behaviour: selection is a tap/click/keypress; hover is decorative only.
  • Result announced via aria-live="polite" after the refresh (see §3).
