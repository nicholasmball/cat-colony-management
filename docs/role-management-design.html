Role management

Design for the Role management feature, building on the approved Step 1 — Requirements & Acceptance Criteria. An admin can change any active member's role between admin, caretaker and feeder directly on the existing /app/members screen — the read-only role pill on each active row becomes an inline <select> + Save, inside a <form action={updateMemberRole}> with a hidden user_id. Confirmed Step 1 guards, mocked explicitly below: admin-only; all three roles assignable; can't change your own row; can't demote the last active admin; active members only (deactivated rows keep a static pill); scoped to the active org, per membership.

Reuse first — no new primitives. The control is built entirely from lib/ui.ts tokens (card, input for the select, btnPrimary for Save, btnGhostDanger for Deactivate, fieldLabel, pill) and the existing SubmitButton / ConfirmButton components. It slots into the Team row that app/app/members/page.tsx already renders — the email + status stay; only the pill is replaced (active rows) or kept (own row / deactivated). The existing ?error= red banner and the green success banner are reused verbatim.

1. Interaction model

Each active member row carries its own little form. The role <select> shows the current role as its resting value and the Save button is hidden until the selection changes — a familiar GitHub/Linear member-row pattern (Jakob's Law): nothing to click means nothing changed. Promotions Save directly. Demotions (any drop in privilege — admin→caretaker/feeder, caretaker→feeder) route through ConfirmButton, so the destructive path takes one extra deliberate tap (Fitts: destructive harder than constructive). The whole thing is a server action — works the same with or without JS; without JS the Save button is simply always visible.

2. Mobile screens (375px)

At 375px the email + status sit on the top line; the role control row (select + Save + Deactivate) stacks beneath, so each control keeps its full ≥44px target without crowding. States shown: resting (no change), changed → ready to save (promote, direct), own row (static, "You"), last admin (guarded), deactivated (static), plus the error and success banners and the demotion confirm dialog.

4. Accessibility notes

  • Every role select has a real <label> — visually-hidden but in the DOM: "Role for ana.feeder@scot.pt", so screen-reader users hear whose role they're editing, not a bare "combobox". Built from the fieldLabel + input tokens.
  • Role is conveyed by text, never colour alone — the pill and the select both spell the word (admin / caretaker / feeder). The accent-tinted pill is decoration on top of the word; legible in greyscale.
  • Keyboard operable end-to-end — Tab to the select, arrow keys / type-ahead to change, Tab to Save, Enter/Space to submit. The ConfirmButton demotion uses the native window.confirm dialog, which is keyboard- and screen-reader-accessible by default.
  • Visible focus ring — the select takes the shared 3px focus:ring-accent/25 from the input token; no focus is ever suppressed.
  • ≥44px targets — the select is min-h-11 (44px), Save is btnPrimary (min-h-11), Deactivate is btnGhostDanger (min-h-11). At 375px they stack so none shrinks below target.
  • No disabled control without an adjacent explanation — the own-row Deactivate and the last-admin demote options are disabled with a visible inline hint ("You can't change your own role" / "promote another admin first") and an aria-describedby tying the select to that hint. The own row simply omits the role control rather than showing a dead one.
  • Banners announce — success reuses okClass with role="status"; errors reuse errorClass with role="alert" (the same ?error= banner the page already renders), so guard failures are read out immediately.
  • Destructive harder than constructive (Fitts) — promote = one tap; demote = an extra confirm step; self-change and last-admin demote have no path at all.

The only new behaviour is the tiny client toggle that reveals Save when the select value differs from the row's current role — a small "use client" wrapper around the existing SubmitButton/ConfirmButton, with graceful no-JS fallback (Save always visible).
