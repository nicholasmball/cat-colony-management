RLS role matrix

Design artefact for the Row-Level Security task. Roles come from memberships(user, org, role); every action is scoped to the caller's own organisation. Deny-by-default — absence of a policy means no access. (Backfilled record; implemented in 0003_rls.sql.)

Within the user's own organisation
Action Admin Caretaker Feeder
Read all org data Yes Yes Yes
Manage colonies / schedules Yes Yes No
Add / edit cats, mark missing Yes Yes Report new only
Record feeds & sightings Yes Yes Yes
Report incidents / new cats Yes Yes Yes
Close / resolve incidents (Q18) Yes Yes No
Edit alert thresholds / urgency Yes Yes No
Invite / manage members & roles Yes No No

Append-only event tables (feeding_events, cat_sightings) are insert-only for any member; deletes are soft (manager UPDATE of deleted_at). Cross-org access is impossible (verified: anon read returns [], anon write returns 401).
