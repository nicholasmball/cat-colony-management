Missing-cat process (human review) — UX Design

SCoT Colony Management App · Step 2 of 6 (UX Design) · Mobile-first PWA · all frames rendered at 375px

Thresholds surface a cat as "of concern"; a human decides what it means. A caretaker reviews each candidate and chooses Ignore, Monitor, or Mark missing — the app never marks a cat missing by itself. Built directly on the just-shipped new-cat review pattern: a per-colony "needs review" section with a count badge, concern-first ordering, and a review action block on the cat detail page. No new visual language.

Scope: this card is in-place surfacing + the caretaker decision only. Alert-engine fan-out (push / SMS / in-app notification when a cat crosses a threshold) is deferred to its own card — nothing here sends a message.

1 · Context & principles

Upstream signal (no change needed): feeders mark cats seen / not_seen / concern in feed-form.tsx on each visit. The threshold engine reads that history and flags candidates using the per-org defaults from the Requirements: cat not-seen = 7 days, repeated not-seen = 3 consecutive visits, plus an explicit feeder "concern" flag. Surfacing a candidate is derived, not stored — exactly like unconfirmed-first ordering today.

  • Never auto-mark (the north star of this card): a threshold only raises a cat for review. The cat stays active until a caretaker explicitly taps Mark missing. Every copy string reflects this — see §3.5.
  • Reuse the new-cat review pattern (Jakob's Law): a per-colony "needs review" section + a count badge + concern-first sort + a review action block on the cat detail page. Caretakers already know this loop from confirming reported cats.
  • Gestalt grouping: "Cats of concern" is its own section, visually separate from the existing "New · unconfirmed" review section and from the plain Cats list. Within it, a "Monitoring" sub-group is split from fresh concerns so the caretaker sees new-since-last-visit work distinctly from cats they already chose to watch.
  • Information scent: each candidate carries a specific reason chip ("Not seen 9 days", "3 missed visits", "Flagged: concern") — never a generic "needs attention".
  • Hick's Law + Fitts: the review block offers three clearly-labelled choices; the two constructive ones (Ignore, Monitor) are reachable side-by-side, the destructive Mark missing is set apart and routed through the ConfirmButton dialog (harder to hit by accident).

2 · How a cat is surfaced (and never auto-marked)

The status pill is honest at every step. A surfaced cat keeps its real active status and gains a review reason; only a caretaker's tap changes the status to missing.

Signal (from feed history) What the app does Cat's stored status
Not seen ≥ 7 days (org default) Surfaces in "Cats of concern" with chip "Not seen N days" active (unchanged)
Not seen on 3 consecutive visits Surfaces with chip "3 missed visits" active (unchanged)
Feeder tapped "Concern" on a visit Surfaces with chip "Flagged: concern" active (the sighting is concern; the cat record stays active)
Caretaker taps Monitor Moves to the "Monitoring" sub-group, keeps watching active (unchanged)
Caretaker taps Ignore Clears from review (false alarm / cat is around); reason resets active (unchanged)
Caretaker taps Mark missing + confirms The one place status changes — by a human, on purpose missing

5 · Interaction & accessibility notes

  • Reason chips — icon + text, never colour-alone (WCAG 1.4.1): "◷ Not seen 9 days" (amber, warn tone), "⚑ Flagged: concern" (red, bad tone — feeder explicitly raised it), "👁 Monitoring since…" all carry a glyph and words. Tone follows statusTone (concern → warn, not_seen → bad), but the words/icon stand on their own in greyscale.
  • Tap targets (Fitts): Ignore / Monitor / Mark missing are ≥48px tall; rows are ≥60px. The destructive Mark missing is positioned apart (its own full-width row, or the right slot when Monitor is gone) and gated behind a confirm dialog — deliberately harder to reach than the constructive actions.
  • Three actions distinguished by label + treatment, not colour (Hick / WCAG 1.4.1): Monitor = primary (accent, the expected next step), Ignore = ghost (neutral), Mark missing = ghost-danger → danger-in-dialog. Each has a distinct verb label; a colour-blind user reads the words and the position, not the hue.
  • Never-auto-mark reassurance in copy: every flagged state says "surfaced for review" / "the app hasn't changed this cat's status" / "A caretaker will review it". The status pill stays active until the human acts; the word "missing" appears only on the action label and after a confirmed decision.
  • Count badges: "Cats of concern" carries an amber badge (live work), Monitoring a parenthetical count, "New · needs review" the existing neutral badge — three queues, three glanceable counts.
  • Keyboard: the Mark-missing dialog (ConfirmButton) traps focus, opens on Cancel, closes on Esc / backdrop; the trigger carries aria-haspopup="dialog".
  • Contrast (WCAG 1.4.3, AA): amber-800 on amber-50 and red-700 on red-50 for chips/banners, emerald-800 on emerald-50 for the empty-state tick, white-on-accent / white-on-red for buttons — all text ≥4.5:1, UI accents ≥3:1.
  • States handled: empty ("No cats need review" + threshold explainer), flagged (all 3 actions), monitoring (Ignore / Mark-missing + "since" indicator), missing (status chip, no actions, attribution), read-only (feeder context, no actions). No loading-skeleton frame here — surfacing is derived server-side on the colony page load, same as unconfirmed-first ordering.
