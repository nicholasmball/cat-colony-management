New cat reporting — UX Design

SCoT Colony Management App · Step 2 of 6 (UX Design) · Mobile-first PWA · all frames rendered at 375px

A feeder-facing quick report so anyone on a colony can flag a cat they don't recognise, without blocking on missing data. Reported cats land as new_unconfirmed for a lightweight, in-place caretaker review. Built entirely on the existing incident-report + segmented-control + confirm-dialog patterns — no new visual language.

1 · Context & principles

Building on the Requirements step: the quick report is distinct from the caretaker "Add cat" form and is available to all roles on the colony page. Required: at least one of name/description. Optional: colour, sex, neutered (tri-state), notes, photo. Photo is non-blocking. Creates status = 'new_unconfirmed'. Caretaker/Admin can Confirm (→ active) or Reject (soft-delete); feeders see read-only.

  • 30-second north star: a feeder in the field types one identifier and taps Report. Everything else is skippable.
  • Recognition over recall & Hick's Law: the report form mirrors the incident form's tile/segment layout the feeder already knows; sex and neutered reuse the exact segmented-control pattern from feed-form.tsx.
  • Information scent: entry point reads "Report a new cat" (not "Add"), the submit reads "Report cat", the review actions read "Confirm" / "Reject".
  • Never block on missing data: only one identifier is validated; a failed photo upload still saves the report (mirrors the incident photo=failed branch).

4 · Interaction & accessibility notes

  • Tap targets (Fitts): every segment button ≥48px tall; text inputs and the photo "Add" button ≥44px; the sticky "Report cat" submit is 52px and full-width. All exceed the 44×44 minimum.
  • Visible labels, never placeholder-only (WCAG 1.3.1 / 3.3.2): each field has a real visible <label>; placeholders only carry examples ("e.g. Smudge"). The eyebrow headings (Sex, Neutered?) label the segmented groups via aria-labelledby.
  • Segmented controls → radiogroup: sex and neutered use role="radiogroup" with three role="radio" buttons and aria-checked, identical to feed-form.tsx / incident-form.tsx. Arrow keys move within the group; the group is one tab stop.
  • Not colour alone (WCAG 1.4.1): the unconfirmed state always pairs the colour with a "★" glyph and the words "New · unconfirmed", plus a left accent border on the list row. Selected segments carry a "✓" in addition to the fill.
  • Contrast (WCAG 1.4.3, AA): accent-green on white and white-on-accent for the primary button, red-700 on red-50 for errors, amber-800 on amber-50 for the photo warning, indigo-on-indigo-50 for the neutral "New" chip — all ≥4.5:1 for text, ≥3:1 for the border/UI accents.
  • Focus order: Name → Description → Colour → Sex group → Neutered group → Notes → Add photo → Report. On a validation error, focus jumps to Name and the alert is announced.
  • Keyboard: the Reject dialog (ConfirmButton) traps focus, opens on Cancel, closes on Esc or backdrop; the trigger has aria-haspopup="dialog".
  • States handled: empty (helper line + example placeholders as onboarding), loading ("Reporting…" disabled submit + "Working…" photo state), error (inline alert that says what to do; photo error inline under the picker), partial (cat saved + amber photo-failed warning — the partial-success path), success (return banner + badged row).

6 · Open question from Step 1 — photo presign key for a feeder report

Recommendation: option (a) — a feeder-allowed, colony-scoped _unassigned presign key, mirroring the incident photo branch.

From a UX standpoint the user must be able to attach a photo before the cat row exists (the photo is part of the 30-second report, taken in the field; the cat id isn't created until submit). The incident form already presigns against a colony-scoped key and then attaches it on save, so reusing that path gives feeders the same proven, non-blocking flow with zero new mental model. The photo lives under the colony's scope (RLS already permits feeders there), and the server action re-keys/links it to the new cat on insert. If linking fails, we fall through to the existing photo=failed branch — the report still saves. Final storage key shape and re-key timing are the engineer's call; this only fixes the requirement that a photo can be captured pre-insert without a separate upload step.
