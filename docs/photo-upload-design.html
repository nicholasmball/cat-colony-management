Photo upload & image pipeline

Building on the Requirements. A reusable image field; first used for the cat photo. Small uploads (client-side resize) → object storage via a presigned URL → thumbnail on the record.

The flow

pick / take a photo→ resize + compress in the browser (~1600px)→ ask server for a presigned PUT URL→ upload straight to R2→ save key + URL on the cat→ thumbnail shows
Storage creds stay on the server — the browser only ever gets a short-lived, single-object upload URL. Key is scoped per org: org/<orgId>/cats/<catId>/<uuid>.jpg.

Where it appears

  • Cat edit + add forms — an image field at the top; optional, never blocks Save.
  • Cat row / detail — small round thumbnail next to the name; falls back to a paw icon when there’s no photo or the image 404s.
  • Component is generic (<ImageUpload owner="cat" id=… />) so colony & incident photos can reuse it later.

Behaviour, permissions & a11y

  • Accepts images only (image/*, capture=camera on mobile); rejects non-images and oversized files with a clear message.
  • Resize/compress before upload keeps field uploads tiny on weak signal and storage small.
  • Presign endpoint authorises the caller (same as editing the record — Caretaker/Admin) and scopes the key to their org.
  • Replace overwrites; Remove clears the reference (old object deleted best-effort).
  • Alt text on the thumbnail (cat name); ≥44px controls; keyboard-operable picker.

Decision for the gate (blocks build)

Storage backend. Plan = Cloudflare R2 (10 GB free, zero egress — per the cost research). That needs you to create a Cloudflare R2 bucket + S3 API token and give me 5 env vars (account id, access key, secret, bucket, public base URL).
Or ship now on Supabase Storage (free 1 GB, zero setup) and migrate to R2 later — trade-off: uses the storage that nudges us toward Supabase Pro sooner.
