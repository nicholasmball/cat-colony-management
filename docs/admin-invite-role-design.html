Step 2 — UX Design & Prototyping

Invite a member directly as Admin

Members screen · app/app/members/page.tsx · SCoT Colony Management (mobile-first PWA)

Feature. Admins can already invite Feeders and Caretakers. This change adds Admin as a third option in the invite form's role <select>, and surfaces a lightweight inline "full access" note when Admin is chosen (per Step-1 AC13 — an inline reveal, not a modal). The screen is already admin-only (page.tsx redirects non-admins), so no new gating is needed. Default role stays Feeder.

Design constraint, restated: this is a single-form change. We reuse the existing design-system tokens (lib/ui.ts: input, fieldLabel, card, btnPrimary) and the existing reveal-on-change idiom already proven in components/role-select-form.tsx (client useState + aria-describedby). No new components, no new round-trip.

1 · Current vs proposed

Today

The role select offers two options; there is no contextual note.

<select name="role" defaultValue="feeder">
  <option value="feeder">Feeder</option>
  <option value="caretaker">Caretaker</option>
</select>

Proposed

Add Admin as the third option. Order follows privilege (Feeder → Caretaker → Admin), reading low-to-high so the most privileged sits at the bottom. Default stays Feeder. An inline note appears only when Admin is selected.

<select name="role" defaultValue="feeder"
        aria-describedby={isAdmin ? "invite-admin-note" : undefined}>
  <option value="feeder">Feeder</option>
  <option value="caretaker">Caretaker</option>
  <option value="admin">Admin</option>
</select>
{isAdmin && <inline note>}

Order rationale. The member-row editor (role-select-form.tsx) lists roles high-to-low (admin, caretaker, feeder) because it edits an existing role. The invite select is a fresh choice that defaults to the least-privileged Feeder, so listing low-to-high keeps the default at the top and makes "stepping up" to Admin a deliberate downward move to the most powerful option. Both reuse the same members.role.* labels — no privilege values are duplicated.

2 · The "full access" note

  • Placement. Inline, directly below the role field, inside the same invite card, spanning the form width (so on the sm:flex-row layout it sits under the email+role row, not squeezed beside a 176px select). It is part of normal document flow — it pushes the Send button down rather than overlaying anything. Not a modal (AC13).
  • Trigger. Pure client-side reveal driven by the select's onChange value (role === "admin"). No extra round-trip — nothing is fetched or posted to show it. Hidden again the instant a different role is chosen.
  • Not colour alone (WCAG 1.4.1). The note pairs an icon marker (a key 🔑 / or an inline SVG shield) with explicit text ("Full access"). Meaning never depends on the amber tint — a greyscale or colour-blind user still gets the icon + the words.
  • Contrast (WCAG 1.4.3). Amber-on-cream note text uses #7a5a16 on #fef6e7 ≈ 6.0:1, comfortably past the 4.5:1 AA threshold for the small text. The 1px #e7c98a border gives a non-colour edge too.
  • Width / 375px. The note wraps cleanly at the smallest target width (see §5 mock). Icon is flex: 0 0 auto so the text reflows beside it without clipping.

3 · States & behaviour

Selected roleNoteNotes
Feeder (default)HiddenInitial render — no note, button sits tight under the row.
CaretakerHiddenNo note for non-admin roles.
AdminVisibleNote reveals inline; aria-describedby ties it to the select.
  • Keyboard. The note is plain inline text, not interactive — it's reached purely by changing the native <select> with keyboard (Tab to select, arrow/type to "Admin"). No focus trap, no extra tab stop, nothing to dismiss.
  • Screen-reader association. When Admin is active, the select carries aria-describedby="invite-admin-note" and the note element has id="invite-admin-note". SR users hear the role label, value, and the full-access description as one unit — exactly the pattern already used for the last-admin hint in role-select-form.tsx.
  • Live announcement (optional, recommended). Give the note container role="status" so AT announces the description when it appears mid-form, without stealing focus.
  • No-JS / progressive enhancement. If JS is unavailable the note simply never shows, but the Admin <option> still submits correctly — the server action is the real authority. The reveal is a non-blocking enhancement, consistent with how the rest of this screen degrades.
  • Form integrity. Showing/hiding the note must not reset the email field or the Send button — it's a sibling element toggled by the same client component that wraps the select; the email input is uncontrolled and untouched.

4 · Proposed i18n keys

One new key per locale. We reuse members.role.admin ("Admin" / "Administrador") for the option label — do not add a second admin label.

KeyEN (messages/en.json)PT (messages/pt.json)
members.invite.adminNote Admins have full access to this organisation — they can manage every colony, member and setting, and invite or remove other admins. Os administradores têm acesso total a esta organização — podem gerir todas as colónias, membros e definições, e convidar ou remover outros administradores.

Placed under a new invite sub-object inside members (sibling of the existing flat keys), keeping the namespace tidy if more invite-specific strings arrive later. If a flat key is preferred to avoid nesting churn, members.inviteAdminNote is the equivalent. Either way the key carries the information scent (it's about the admin invite note) and both locales must stay key-parity (guarded by messages/messages.test.ts).

6 · Accessibility checklist (this change)

  • 1.4.1 Use of Colour — note conveys meaning via icon + "Full access" text, never colour alone.
  • 1.4.3 Contrast — note text ≈ 6.0:1 on its amber background; visible 1px border edge.
  • 1.3.1 / 4.1.2 Name, Role, Value — aria-describedby links select → note so the description is part of the field's accessible name when Admin is active.
  • 4.1.3 Status Messages — role="status" on the note announces the reveal without moving focus.
  • 2.1.1 Keyboard — entirely operable via the native select; the note adds no tab stop and needs no dismissal.
  • 2.5.5 Target Size — select and Send button keep the ≥44px min-height from input / btnPrimary; the note is non-interactive.
  • 3.2.2 On Input — changing the role only reveals/hides adjacent text; no navigation, no submit, no context change.
  • Reduced motion / no-JS — instant show/hide (no animation required); Admin still submits without JS, server action enforces authority.

Implementation footprint (for Step 3). Add <option value="admin"> to the invite select; lift that select into a tiny client wrapper (mirroring role-select-form.tsx) that tracks the selected value and conditionally renders the note with id="invite-admin-note" + aria-describedby; add members.invite.adminNote to en.json and pt.json. No server-action change is required for the note itself — the existing invite action already accepts any valid role; confirm it permits admin (Step-1 AC: admins may invite admins).
