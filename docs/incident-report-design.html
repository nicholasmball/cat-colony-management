Incident reporting — create flow

Step 2 (UX) of the "Incident reporting (create)" task. Step 1 (Requirements & AC) set the north star: a Feeder reports an incident in seconds in the field — the "30-second update" applied to problems. Only the incident type is truly required (the colony comes from the route). Everything else — urgency (defaults to the org's Not urgent), cat, notes, photo — is optional or pre-defaulted, honouring CLAUDE.md's rule: "accept incomplete data — never block creation because a field is missing." This document covers the CREATE flow only; the list / triage / resolution view, the alert engine, and settings are out of scope.

House style reused, no new primitive. This screen is a sibling of components/feed-form.tsx: a colony-scoped quick form with a single sticky accent CTA. The type picker is the same selection-button family as the sighting tri-toggle (outline → fill); the urgency control is the exact segmented pattern just shipped for "Was the colony fed?" (docs/feeding-update-redesign.html); cat rows reuse the avatar row from app/app/colonies/[id]/page.tsx; the photo block reuses components/image-upload.tsx. Tokens from lib/ui.ts (btnPrimary, input, card). Route: app/app/colonies/[id]/incidents/new.

The feeder is already on the colony they're standing in. 2. The type picker — Hick's Law on 9 options

The hardest UX call. Nine fixed types is a lot for "in seconds" (Hick's Law: choice time rises with option count). We don't drop any — they're a fixed enum — but we tame the scan cost four ways:

  • Group & order by severity. A small Urgent / danger header floats the four serious, time-critical types to the top (poisoning, sick/injured, dog danger, threat from person) — the ones where seconds matter. A neutral Reports group holds the rest (new cat, missing concern, dead cat, feeding/access problem, other). Two short lists scan far faster than one flat nine (chunking).
  • Icon + label, never icon-alone. Each button pairs a glyph with its word, so it's recognisable at a glance and unambiguous (recognition over recall; never colour/icon-alone).
  • 2-up grid, large targets. Each tile is ≥56px tall and a full half-width — thumb-friendly, no mis-taps, no horizontal scan across a wide row.
  • Single-select, clear chosen state. Picking one fills it accent with a ✓; the rest stay outline. The required field is obvious and self-correcting.

6. Accessibility notes (§)

  • § Type picker = single-select radiogroup. The 9 tiles are one role="radiogroup" labelled "What's happening?" with each tile a role="radio" / aria-checked and aria-required="true". The two group dividers ("Urgent / danger", "Reports") are visual chunking only — exactly one radio is checked across the whole group.
  • § Urgency = the shipped segmented pattern. Two segments, role="radiogroup" labelled "Urgency", each a role="radio" / aria-checked — identical semantics to the "Was the colony fed?" control. Exactly one selected at all times; Not urgent is the default aria-checked segment.
  • § Never colour-alone. Urgent isn't only red — it carries a ⚠ glyph + bold weight + fill, plus the live hint text changing to "Caretakers will be alerted". A selected type tile shows a ✓ + fill + accent border, not hue alone. Legible in greyscale and to colour-blind users.
  • § ≥44px targets. Every type tile (≥56px), both urgency segments (≥48px), every cat row (≥56px), the photo "Add photo" button, and the submit are all ≥44px — the 2-up grid holds full-width hit areas at 375px with no overflow.
  • § Visible focus. A focus-visible accent ring on every control, the same the design system already applies (focus:ring-accent/25 in lib/ui.ts).
  • § Keyboard order. Linear: Tab → Type group (one stop, ←/→ between tiles) → Urgency (one stop, arrows) → Cat group → Notes → Add photo → Report incident. The submit is the only control that submits.
  • § Visible labels, not placeholders. "What's happening?", "Urgency", "Which cat? (optional)", "Notes (optional)", "Photo (optional)" are real on-screen headings — recognition over recall, the required one marked with a visible * and aria-required.
  • § Errors & status announce. The "choose a type" message is role="alert" (assertive); the offline/queued and photo-failed notices are role="status" (polite) so they announce without yanking focus from a saved report.
