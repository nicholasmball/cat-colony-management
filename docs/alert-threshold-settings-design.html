Alert threshold settings

A small, manager-editable form for the three alert thresholds that drive the dashboard and the alert engine: when a cat is flagged not seen, when a run of misses becomes repeated not-seen, and how long after a feeding window a feed counts as missed. Mobile-first (375px), WCAG 2.1 AA, EN + European Portuguese.

Backend is already in place — no migration. The alert_settings table (1:1 per org) holds not_seen_days (default 7), repeated_not_seen (default 3) and feeding_missed_hours (default 12), and a row is auto-inserted on org creation (supabase/migrations/0002_domain.sql). Crucially, RLS "managers update alert_settings" (0003_rls.sql) already grants both Admin AND Caretaker (the manager role) write access. So the entire gap is UI placement — there is no schema or policy work to do.

Placement recommendation

The card asks for caretakers to edit thresholds, but the Org settings page (app/app/org/page.tsx) hard-redirects anyone who isn't an admin (if (org.role !== "admin") redirect("/app")) and its nav.org entry only appears in adminItems (lib/nav-items.ts). Caretakers literally cannot reach it. The three options considered:

Option What it is Verdict
(a) Dedicated manager-gated page
/app/alerts 		Its own route gated to {admin, caretaker} (same guard the dashboard already uses), linked from the manager nav and from the dashboard's "Cats not seen / concern" card. Recommended Both roles reach it, clean role boundary, reuses the existing form pattern verbatim, keeps the admin-only Org page untouched.
(b) Split the Org page Show the thresholds card to managers while name/notes/timezone stay admin-only on the same page. Avoid Forces per-section gating inside one page + relaxes the nav.org gate; most complex, and caretakers would see a page titled "Organisation" they mostly can't use.
(c) Dashboard collapsible section Embed the form on /app/dashboard where managers already land. 2nd choice No nav change needed, but mixes a low-frequency settings write into a read-only oversight surface and complicates the dashboard's all-clear / data-fetch flow.
Decision: Option (a) — a dedicated /app/alerts page gated to {admin, caretaker}. It is the cleanest fit: it reuses the Org-page form pattern (card / fieldLabel / input / SubmitButton, ?error= & ?saved=1 toast) with zero new UI primitives, gives both manager roles a stable home for alert tuning, and leaves the admin-only Org page exactly as it is.

Entry-point change: add one item to the managerItems array in lib/nav-items.ts — { href: "/app/alerts", labelKey: "nav.alerts", exact: false } — so it appears for admin and caretaker (it sits in managerItems, not adminItems). Add a secondary deep-link from the dashboard "concern" card (e.g. "Adjust alert thresholds →") so managers can jump straight from where they notice the flags. The page guard mirrors the dashboard: if (org.role !== "admin" && org.role !== "caretaker") redirect("/app/today").

Accessibility checklist (WCAG 2.1 AA)

  • ✓ Visible <label> on every field (not placeholder-only); hint sits between label and input.
  • ✓ type="number" + inputmode="numeric" → numeric keypad on mobile.
  • ✓ Hint and any error tied to the input via aria-describedby="ns-hint ns-err"; invalid fields get aria-invalid="true".
  • ✓ Errors convey meaning by icon + text, not colour alone (1.4.1); red border is a secondary cue.
  • ✓ Error banner is role="alert"; saved toast is role="status" (polite).
  • ✓ Inputs and the Save button are ≥44px tall (Fitts); focus ring is a 3px accent halo at ≥3:1 against the surface.
  • ✓ Body/label text on surface ≥4.5:1; error red #b91c1c on white ≈ 5.9:1.
  • ✓ Fully keyboard-operable; logical tab order top-to-bottom ending on Save.
  • ✓ Server validation mirrors the client range checks (1–60 / 1–10 / 1–72) so JS-off and tampered submits are still safe.
